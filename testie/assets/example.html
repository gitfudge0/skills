Test Blueprint

Dynamic lockscreen accent color

Extracts a dominant color from the now-playing track's album art and applies it as the accent color across the MusicLock lockscreen widget, replacing the fixed default accent.

Risk: High 11 cases · generated from diff review

What this touches

The extraction logic is new and isolated (bitmap in, color out), but it sits directly in the lockscreen render path, so a slow or crashing extraction affects something users see many times a day, not just once. It depends on Android's Palette API and on album art actually being embedded in the track's metadata, both of which are outside this code and known to be inconsistent in practice. It's also gated by the existing "dynamic theming" setting, so the off-state has to keep working exactly as before. Nothing here touches playback, storage, or other screens.

Test strategy

3Critical

Weighted toward unit tests because the actual risk is concentrated in the extraction function itself — mainly what it does with album art that doesn't cooperate (missing, monochrome, low-contrast). Integration and e2e cases are kept to the handful of places where extraction results actually cross into rendering: track changes, the theming toggle, and a failure not being allowed to take the rest of the lockscreen down with it.

U Unit — color extraction

U1 Extracts a usable accent color from typical album art Critical

Run extraction against a set of representative real album art (vivid cover, muted cover, photo-based cover).

ExpectedReturns a non-null color for every sample, and the returned hue is visibly related to the artwork's dominant color, not a generic fallback.

U2 Falls back cleanly when artwork has no clear dominant color High

Run extraction against flat grey/monochrome or near-white album art.

ExpectedReturns the defined default accent color rather than a near-invisible or undefined one — no exception thrown.

U3 Handles missing artwork without crashing High

Run extraction on a track with no embedded artwork (null/empty bitmap input).

ExpectedReturns the default accent color immediately; no null-pointer exception, no blocking call to a Palette instance that doesn't exist.

U4 Extracted color meets minimum contrast against the lockscreen background Medium

Run extraction against artwork whose dominant color is very close in luminance to the lockscreen background (e.g. a near-black cover on a dark lockscreen).

ExpectedThe color returned for display is adjusted to meet a minimum contrast ratio, rather than producing text/accents that are effectively invisible.

U5 Extraction result is cached per track Low

Call the extraction function twice in a row for the same track.

ExpectedThe second call returns the cached result rather than recomputing — verified by call count on the underlying Palette invocation, not by timing.

I Integration — wiring into the lockscreen

I1 Track change triggers a new extraction and theme update Critical

Play track A, let the lockscreen settle, then switch to track B with visibly different album art.

ExpectedThe lockscreen accent updates to match track B within one render cycle — no stale color left over from track A.

I2 Dynamic theming toggle off bypasses extraction entirely High

Disable the "dynamic theming" setting, then play a track with vivid album art.

ExpectedLockscreen uses the fixed default accent; extraction is not invoked at all (verify via call count, not just visual output — this also protects the battery/perf case).

I3 Extraction failure doesn't block the rest of the lockscreen Medium

Force the extraction call to throw (simulate corrupted artwork bytes).

ExpectedLockscreen still renders fully with the default accent — track title, controls, and artwork placeholder all still appear; the failure is contained.

E End-to-end — real usage

E1 Playing a song with vivid art visibly re-themes the lockscreen Critical

From a cold app state, lock the phone and start playback of a track with distinctive album art.

ExpectedThe lockscreen accent visibly reflects the artwork within a perceptibly instant timeframe — no flash of default color that lingers.

E2 Rapid track-skipping settles on the final track's color without flicker or crash High

Skip through 5+ tracks in quick succession (faster than each extraction likely completes).

ExpectedNo crash, no visible rapid color-flicker, and the lockscreen settles on the correct color for whichever track is actually playing once skipping stops — not a color from a track that was skipped past.

E3 Disabling dynamic theming mid-session reverts immediately Medium

While a track with an extracted accent color is playing and visible on the lockscreen, go into settings and turn dynamic theming off.

ExpectedLockscreen reverts to the default accent immediately, without needing the app or device to restart.

Explicitly not covered here

  • Whether the extracted color is the "best" or most aesthetically pleasing choice from a given artwork — that's a design/subjective judgment, not a pass/fail QA concern.
  • Performance under extreme low-memory or low-end-device conditions — deferred to a dedicated perf pass rather than folded into this feature's functional tests.
  • Correctness of the Android Palette API's internal color-extraction algorithm — that's Android platform code, not ours to test.
  • Any existing lockscreen layout/control behavior untouched by this change — covered by the existing suite already, not re-verified here.
