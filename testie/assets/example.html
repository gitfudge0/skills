AVA-170Test Plan.

A risk-first plan for the AVA-170 report-UI fixes: regression coverage on the exact reported symptoms, unit tests pinning the applyEnabled boundary conditions, and integration/e2e checks confirming the fixes hold end-to-end. Critical paths guarded first, edge cases close behind.

Unit tier (7 cases, U1–U7) pins the applyEnabled boundary conditions and the two supporting-copy fixes so a future refactor can't silently reintroduce the stuck-Apply bug.

Three flows are worth a human pass before merge: default-preset Apply in the Run Now modal, the Interactions search placeholder copy, and the Between field's custom DatePicker/TimePicker click affordance.

Three coverage gaps identified — unexecuted e2e specs, no direct applyEnabled test (now closed), orphaned equality helper (now removed).

1.

Stuck Apply on re-selected preset

Run Now / Quick Download modals defaulting to last30days kept Apply disabled even when re-selecting the already-applied preset.

2.

Interactions search copy mismatch

Placeholder/aria-label read phone-only despite the field actually matching by phone number or Call ID.

3.

Between field click affordance

Native datetime inputs were replaced by the app's custom DatePicker with an embedded TimePicker; clicking anywhere in the field now opens it (value format unchanged).

KnownGaps.

!

1. Nothing has actually been run

Both new unit test files' coverage of these three fixes is inferred from reading the diff and existing specs, not from a green test run — CLAUDE.md's Task Completion Checklist requires bun run lint, bun run build, and bun run test:run before calling work done, and the e2e specs for this branch have not been executed against the docker environment. Until that happens this is a plan, not a verified pass.

2.

No unit test for applyEnabled directly

date-range-picker.test.ts only tests the two exported pure functions (getPresetCalendarHighlight, areDateRangePickerSelectionsEqual); applyEnabled itself is an internal useMemo with no direct unit coverage — U1–U4 above would need a rendered-component test (React Testing Library) asserting the Apply button's disabled state, since the logic isn't exported as a standalone function today.

3.

Dead/orphaned helper

areDateRangePickerSelectionsEqual is exported and still has its own passing unit tests, but after this fix nothing in date-range-picker.tsx calls it. Left as-is, it's a maintenance trap: a future reader may assume it still gates Apply and "restore" the old buggy check. Either remove it (and its now-orphaned tests) or leave a comment explaining why it's kept for potential reuse.

Out of scope
  • Visual/pixel-level verification that the click affordance of the custom DatePicker/TimePicker looks right — that's a design judgment call, better suited to the repo's careconvoy-visual-verify skill than a pass/fail test case.
  • Cross-browser rendering quirks of the custom DatePicker/TimePicker component library itself (Safari/Firefox quirks) — that's a third-party component concern, not code this branch owns; the fix no longer depends on the native showPicker() API.
  • Complaints page's underlying search-matching logic (phone-only, no Call ID match) — explicitly untouched by this branch per the ticket, and not re-verified here.
  • Report execution correctness once a date range or Between filter is actually applied (query building, result correctness) — this plan covers only the UI-interaction bugs reported in AVA-170, not report data correctness, which is pre-existing and unrelated.
